# White-Noise Falling-Block Game
## Technical Design Document v0.1

### 1. Purpose

This document describes the technical design of a falling-block game whose gameplay foundation is a TAP/TGM2-ARS-inspired implementation, and whose artistic identity is the white-noise theme.

The central design requirement is a deliberate duality:

- In implementation terms, **solid mode** and **noise mode** must differ only by theme materials and not by gameplay logic.
- In experiential terms, the **white-noise theme** is the defining identity of the project.

The engine must therefore be built as one coherent gameplay system with one coherent motion/presentation model, while allowing the white-noise theme to transform the player's perception of that system.

---

### 2. Project Principle

The most important principle of the project is this:

**Noise mode is architecturally small, but conceptually central.**

That means:

- The game must not be implemented as a normal game plus a special-case noise variant.
- The game must not be documented as though noise were an afterthought.
- The correct architecture is one engine, one motion model, one rendering model, and multiple themes.
- The white-noise theme is the artistic center even though it should be a minor switch at the code level.

This principle should influence all technical decisions.

---

### 3. Gameplay Baseline

The gameplay baseline is:

- TAP / TGM2-style ARS
- 3 Next previews
- no Hold
- no T-Spin handling
- Up = hard drop + immediate lock (custom deviation)
- Down = TAP/TGM2-style soft drop behavior (no extra custom lock rule)
- custom piece-count-based gravity progression
- custom score based on piece count, cleared lines, and bravo
- line clear animation and impact shake are custom presentation behaviors
- separate user HUD and debug HUD

The project is not trying to reproduce every historical detail exactly, but it is trying to stay disciplined and reference-driven rather than drifting into generic "Tetris-like" behavior.

---

### 4. System Overview

The project should be divided into the following high-level systems:

1. **Gameplay Core**
   - Pure rules and state transitions
   - No knowledge of textures, noise, or colors

2. **Presentation State**
   - Shared motion and animation model
   - Active piece movement, entry motion, queue motion, impact shake, line clear motion
   - Common across solid and noise themes

3. **Theme / Material System**
   - Supplies visual materials for all roles
   - Determines whether the game appears in solid mode or noise mode

4. **Renderer**
   - Draws gameplay state using presentation state and theme materials
   - Same rendering structure for all themes

This separation is mandatory. Theme differences must never leak into gameplay logic.

---

### 5. Core Architectural View

The game should not be conceptualized as drawing symbolic blocks directly from logical occupancy every frame.

Instead, the correct visual model is:

- the **active piece** is a movable visual layer with a persistent material
- the **settled field** is a composited surface
- **locking** commits the active piece into the settled field surface
- the **Next queue** displays persistent materials and animates them within its own panel
- the **frame**, **background**, and **panels** are also persistent visual roles owned by the theme

This visual model must be shared by both solid mode and noise mode.

In other words, the difference between the two modes is not in how the game is drawn, but only in what materials are assigned to the visual roles.

Tetromino materials should be treated as continuous piece surfaces rather than as repeated one-cell tiles. A rendered piece should conceptually own one piece-surface image, and each occupied cell should sample its corresponding region from that surface.

---

### 6. Theme System

#### 6.1 Theme abstraction

A theme is a complete assignment of materials to visual roles.

Visual roles include at minimum:

- background
- frame
- playfield surface
- Next panel background
- each tetromino type
- cleared-line presentation layer
- optional UI text and accents

Each role owns a persistent material.

Scene-surface roles such as the background, frame, playfield surface, and side panels should be represented as full-surface textures or images rather than repeated tiles. This keeps the renderer compatible with future arbitrary theme artwork and avoids tiling periodicity that would weaken the noise concept.

Tetromino roles should likewise be represented as persistent piece surfaces. The renderer should rotate and sample those surfaces by piece orientation rather than stamping the same one-cell tile repeatedly.

#### 6.2 Solid theme

In solid mode:

- tetromino materials may be flat colors or simple visible textures
- the frame and background may be plain and readable
- visibility is intentionally high

Solid mode exists primarily for development, tuning, debugging, and gameplay validation.

#### 6.3 Noise theme

In noise mode:

- all major visible elements use fixed white-noise materials
- the noise must be pre-generated and persistent
- the generated noise should be true binary black/white noise rather than grayscale noise
- the default generated noise should use display-space grain rather than raw backing pixels; the current target is `1 x 1` CSS-pixel blocks
- the frame, playfield surface, active piece, and settled field should converge toward the same perceived gray density when static
- each tetromino material should be generated as one continuous piece surface rather than as four repeated cell patches
- the `NEXT` helper label should be omitted
- user-HUD labels and digits should render as glyph-only overlays rather than as boxed widgets
- the player should rely on motion and change rather than static contrast

Noise mode is the artistic identity of the project.

#### 6.4 Theme invariants

Switching themes must not alter:

- piece rules
- spawn rules
- randomizer
- timings
- movement behavior
- lock behavior
- queue behavior
- animation behavior
- state transitions

If gameplay changes when the theme changes, the architecture has failed.

#### 6.5 Theme selection during development

Theme selection is a development concern, not a gameplay mode.

Rules:

- the application should support hidden theme selection between `solid` and `noise`
- a dev-only Vite environment value such as `VITE_THEME` may choose the theme at startup
- a hidden persisted storage override may still exist as a fallback
- if both environment and storage are present, the environment value takes precedence
- in shipped `normal` mode, the game should always boot into the `noise` theme regardless of debug theme overrides
- hidden theme overrides exist for debug use, not for the normal player-facing boot path
- theme selection must not alter gameplay state, randomization, or timings
- in `debug` and `debug20g`, the active theme may be toggled live without resetting gameplay or presentation state
- because debug modes can toggle theme at runtime, no additional combined startup modes are required

### 7. Material Model

The project should treat materials as persistent assets attached to roles rather than one-frame drawing instructions.

This means:

- each tetromino type has a persistent material
- an active piece instance is a transform applied to its type material
- the settled field is a composited accumulation of prior piece materials
- the Next queue displays the same materials used by active pieces
- the frame and panels use theme-defined persistent materials

For tetrominoes, the persistent material should be modeled as one piece surface rather than as a reusable one-cell tile. Rendering a cell should mean sampling the appropriate region from that piece surface.

#### 7.1 Material rotation rule

Material rotation is a presentation rule, not a gameplay rule.

- `I`, `J`, `L`, `S`, `T`, and `Z` rotate their visual material together with the piece
- `O` does not rotate its visual material
- this applies even if the `O` material is not rotationally symmetric

This rule is especially important for the noise theme. It keeps `O` visually stable while allowing other pieces to behave like rotated visual objects.

This model applies equally to solid and noise themes.

Examples:

- in solid mode, the T piece material might simply be a magenta fill
- in noise mode, the T piece material might be a fixed white-noise image

But in both modes, it is still the same role in the same pipeline.

### 8. Gameplay Core

The gameplay core should contain only deterministic game logic.

Responsibilities:

- field occupancy
- active piece state
- queue state
- spawn and game-over checks
- gravity
- lock delay
- movement validity
- rotation validity
- grounded detection
- line clear detection
- ARE and other transition states
- scoring

The gameplay core must not know whether the piece is drawn as a colored shape or as white noise.

#### 8.1 Rules reference

The gameplay core should follow a TAP/TGM2-inspired baseline with explicit defaults:

- ARS (not World/SRS)
- no Hold
- no T-Spin logic
- step-reset-only lock behavior
- one-row buffer zone above the visible 20-row field (effective logical height 21)
- deterministic frame-based timing at 60 fps
- custom speed progression that is not TAP section logic

#### 8.2 ARS behavior requirements (TAP-style)

- spawn in the upper rows used by ARS/TAP (piece appears high, not guideline-style low spawn)
- L/J/T initial orientation follows ARS (initial flat side facing up)
- I/S/Z functionally use two orientations in play
- kick order for non-I pieces: base rotation -> +1 right -> +1 left
- I piece does not wall-kick in TAP-style ARS
- L/J/T center-column constraint must be respected (no generic SRS kick table)
- no world-rule floor-kick behavior
- no movement-reset or Infinity-style lock behavior

#### 8.3 Randomizer requirements (TGM2/TAP)

- use TGM history randomizer
- history size = 4
- reroll attempts per piece = 6
- first piece may not be S, Z, or O
- initial history seed is TAP-style (`Z, Z, S, S` behavior)

#### 8.4 Deliberate deviations from original TAP

- preview count is `3` (TAP arcade baseline is `1`)
- `Up` is hard drop with immediate lock (TAP sonic drop does not lock)
- level progression is custom and based on locked piece count rather than TAP sections

No other gameplay-rule deviations from TAP/TGM2 should be introduced without an explicit design update.

#### 8.5 Core state model

The top-level gameplay state should be exactly one of:

- `ARE`
- `Spawning`
- `Active`
- `LineClear`
- `GameOver`

`Grounded` is not a separate top-level state. It is a condition of the `Active` state.

Additional state requirements:

- the queue always contains enough pieces to show 3 previews and produce the next active piece
- spawn failure is checked only during `Spawning`, after allowed spawn-time inputs are applied
- locking inside the single hidden buffer row is legal
- blocks above the hidden buffer row are outside the recoverable field

`Pause` is not a gameplay-core state. It should remain an application-level run-state that suspends simulation without changing the gameplay state machine.

#### 8.6 Scoring requirements

Scoring is intentionally simpler than TAP.

Rules:

- a piece that clears `0` lines awards `0` score
- base line-clear values are:
  - `1 line = 100`
  - `2 lines = 300`
  - `3 lines = 500`
  - `4 lines = 800`
- score scales by piece count using:
  - `levelMultiplier = (100 + piecesBeforeLock) / 100`
- a bravo applies a `x4` multiplier
- if no manual shake has been used since the previous line clear, the next line clear receives a `x2` no-shake reward
- using manual shake removes that reward for the next line clear only; after a line clear resolves, the reward state resets
- combo, soft-drop, hard-drop-distance, sonic-drop, and speed bonuses do not exist

Recommended formula:

`scoreGain = floor(clearBase[linesCleared] * levelMultiplier) * bravoMultiplier * noShakeMultiplier`

This keeps score readable and arcade-like without inheriting TAP's full scoring identity.

### 9. Input System

The input system should be designed around frame-based evaluation with support for buffered and preloaded actions.

Responsibilities:

- current button state
- pressed-this-frame transitions
- DAS charging and repeat behavior
- rotation input handling
- hard drop input
- soft drop input
- pre-entry lateral input intent
- stored IRS intent

A key project requirement is that player intention can exist before the next piece becomes active.

If Left or Right is already being held during entry delay, the next piece should enter according to that stored horizontal intent rather than appearing as a neutral centered object and moving only afterward.

This behavior is important for two reasons:

- it supports the TGM-family feel
- it strengthens the noise concept by giving the new piece motion at the moment of entry

#### 9.1 Input rules

- Left / Right during `ARE` and `LineClear` continue charging DAS
- rotation input during `ARE` and `LineClear` is stored as IRS intent
- `Up` and `Down` during `ARE` and `LineClear` do not create deferred drop actions
- if both horizontal directions are held, the latest pressed direction wins
- during spawn processing, apply IRS first and lateral intent second
- if a spawn-time adjustment is invalid, it fails normally rather than invoking non-ARS rescue logic

#### 9.2 Application pause control

- the application should expose a pause toggle on `P`
- pause is outside gameplay logic and should not be represented as a gameplay phase
- while paused, gameplay simulation and presentation timers stop advancing
- the current rendered frame remains visible and a pause overlay may be shown
- unpause resumes from the same gameplay state with no accumulated simulation backlog
- the pause overlay should use the same textured label-rendering system as the HUD
- the pause prompt should be split across two lines (`PRESS P` / `TO RESUME`) rather than shrinking the text below the HUD label scale

#### 9.3 Application boot modes

The application should support three boot modes:

- `normal`
- `debug`
- `debug20g`

Rules:

- `normal` is the user-facing mode and starts unpaused
- `debug` is non-user-facing and starts paused
- `debug20g` is non-user-facing and starts paused
- `normal` should seed the randomizer from a time-derived source rather than a fixed development seed
- `debug` and `debug20g` should use a fixed deterministic seed by default so behavior stays reproducible during testing
- after a game-over restart, the next run should start unpaused regardless of debug boot mode
- `debug20g` should initialize directly into `20G` conditions with piece count set to `500`
- no in-game mode transitions are required
- debug modes should be entered only through a hidden startup mechanism rather than a visible menu

Development boot mechanism:

- the application may read a dev-only startup mode from the Vite environment via `VITE_BOOT_MODE`
- the development scripts should include direct entry points for `debug` and `debug20g`
- a hidden persisted storage override may still exist as a fallback
- if both environment and storage are present, the environment value takes precedence

### 10. Timing and Simulation

The simulation should run at a fixed 60 fps.

All timing-sensitive behavior should be frame-based and configurable.

At minimum, the configuration system should support parameters for:

- gravity
- soft drop speed
- lock delay
- ARE
- line clear delay
- DAS
- repeat interval / ARR equivalent if needed
- entry animation duration
- lateral movement animation duration
- queue slide duration
- line clear slide duration
- impact shake duration
- impact shake amplitude

Even if early versions use provisional values, the architecture should assume these values are tunable data.

#### 10.1 Frame step order

Each simulation frame should resolve in this order:

1. read current input state
2. update edge-triggered input events, DAS state, and stored IRS intent
3. advance the gameplay state machine
4. update presentation state from finalized gameplay state

Gameplay must remain deterministic and presentation must never feed rules back into the simulation.

#### 10.2 Active-piece rule order

While in the `Active` state, the frame order should be:

1. rotation input
2. horizontal movement input and DAS repeat
3. `Up` hard drop if pressed this frame
4. `Down` soft drop / grounded lock behavior
5. gravity
6. grounded detection and lock-delay evaluation

This ordering keeps player action primary and avoids gravity consuming a frame before input is resolved.

#### 10.3 Movement, drop, and lock semantics

- rotation uses TAP-style ARS only
- non-`I` rotation test order is: basic rotation -> `+1` right -> `+1` left
- `I` never kicks
- successful shifts and rotations do not reset lock delay
- only an actual downward step of at least one row resets lock delay
- `Down` accelerates descent while airborne
- the current default soft-drop speed is `256` internal gravity (`1G`), intentionally gentler than a harsher TAP-like fast-drop feel
- if `Down` is newly pressed while the piece is already grounded, it may force immediate lock
- `Up` moves the piece to its final reachable resting position and locks it immediately in the same gameplay step
- `Up` grants no post-drop lock delay and no post-drop correction window
- `S` triggers manual shake and marks the current scoring interval as having used shake
- zangi-style play enabled by sonic drop should not exist in this project

#### 10.4 State transition details

`ARE`

- no active piece exists
- the ARE timer counts down once per frame
- DAS and IRS may continue charging / buffering
- when the timer reaches zero, transition to `Spawning`

`Spawning`

- dequeue the next piece and immediately refill the preview queue from the randomizer
- create the piece using ARS spawn position and orientation
- apply stored IRS intent first
- apply pre-entry lateral intent second
- if gravity is `20G`, settle the piece immediately to its final resting height before entering `Active`
- if the resulting spawn state is invalid, transition to `GameOver`
- otherwise transition to `Active`

`Active`

- the active piece owns type, position, orientation, gravity accumulator, grounded flag, and lock-delay timer
- below `1G`, gravity uses fractional accumulation
- at `1G` or more, the piece descends as far as allowed by collision
- at `20G`, newly spawned pieces should already be settled when `Active` begins
- at `20G`, there is no airborne traversal phase between spawn and grounded play
- grounded manipulation still exists after settle and before lock resolves
- on the first grounded frame, start the lock-delay timer
- while still grounded, decrement lock delay once per frame
- if lock delay reaches zero, lock immediately

`Lock resolution`

- merge the piece into the logical field immediately
- increment the locked-piece counter immediately
- detect completed lines immediately
- if no lines were cleared, start normal `ARE`
- if one or more lines were cleared, transition to `LineClear`

`LineClear`

- no active piece exists
- gameplay is paused except for input buffering, DAS charging, and IRS storage
- cleared rows remain logically frozen during the line-clear delay
- the field collapses exactly when the line-clear delay ends
- after collapse, transition to `ARE`

`GameOver`

- game over occurs only when spawning the next piece fails after allowed spawn-time processing
- the blocked spawn piece should remain visible even if it overlaps existing stack cells

#### 10.5 Project default timing profile

The project should use fixed TAP-like handling delays and a custom gravity ladder.

Default handling delays (frames):

| Parameter | Value |
| --- | ---: |
| ARE | 16 |
| Line ARE | 12 |
| DAS | 16 |
| Lock delay | 30 |
| Line clear delay | 12 |

These values stay fixed across the whole game unless explicitly re-tuned later.

This timing profile intentionally pulls the post-lock cadence closer to early TAP Death rhythm while keeping the project-specific progression model and preserving a full 30-frame lock delay.

#### 10.6 Custom gravity progression

Gravity progression is based on the number of pieces locked, not TAP sections or level stops.

Rules:

- gravity is a monotonic function of locked piece count
- gravity is the only timing value that changes as progression advances
- before `20G`, the maximum gravity should be approximately `5G`
- at `500` locked pieces, gravity jumps to `20G`
- reaching `20G` does not shorten lock delay, ARE, DAS, or line clear delay

Recommended default formula:

For `pieceCount < 500`:

- `p = pieceCount / 499`
- `g0 = 4 / 256`
- `gravityG = g0 + (5 - g0) * p^2`
- `internalGravity = round(gravityG * 256)`

For `pieceCount >= 500`:

- `internalGravity = 5120` (`20G`)

This curve is intentionally inspired by the broad feel of TGM/TAP acceleration without copying TAP's exact gravity table or its non-monotonic dips.

Useful checkpoints:

| Locked pieces | Approx gravity |
| --- | ---: |
| 0 | 0.015625G |
| 100 | about 0.22G |
| 200 | about 0.82G |
| 300 | about 1.82G |
| 400 | about 3.22G |
| 499 | about 5G |
| 500+ | 20G |

### 11. Motion / Presentation State

The presentation state is shared by all themes.

It exists to bridge the gap between logic and rendering while remaining theme-agnostic.

Responsibilities include:

- active piece display transform
- active piece material rotation transform
- entry motion state
- lateral movement interpolation state
- gravity display motion state
- hard drop motion state if any visual treatment is desired
- queue slide state
- line clear slide state
- impact shake state
- screen or frame shake offsets

This layer should contain the answer to questions such as:

- Is lateral movement animated or instant?
- Does the queue slide or snap?
- How long does impact shake last?
- How far do cleared lines move before disappearing?

These behaviors must be shared across solid and noise themes.

---

### 12. Renderer

The renderer should consume:

- gameplay state
- presentation state
- theme materials

The renderer should not invent gameplay behavior.

Its job is to visualize already-determined state using the theme's materials and the common presentation rules.

The renderer should support the following visual structure:

- background layer
- frame layer
- playfield surface layer
- settled field layer
- active piece layer
- Next queue layer
- user HUD layer
- debug HUD layer
- line-clear temporary layer
- shake transform applied at the appropriate visual scope

The same renderer structure should be used for both themes. Theme switching should only replace owned materials and textures.

Renderer requirements for piece materials:

- non-`O` pieces apply piece rotation to their material when drawn
- `O` keeps a fixed material orientation regardless of gameplay rotation state

The renderer should allow theme switching without changing draw order or gameplay behavior.

Frame requirements:

- the playfield frame is a first-class textured role rather than a stroked debug outline
- the frame is exactly one cell thick on all four sides
- its outer size is the playfield size plus two cells in both width and height
- it should be rendered as a textured ring surrounding the playfield

#### 12.1 HUD requirements

The project should distinguish between a user-facing HUD and a debug HUD.

Application-shell layout requirements:

- the game should use the browser viewport from near the top rather than reserving a title/header strip above the canvas
- decorative top-level labels such as the game title or current-theme subtitle should not consume vertical play space
- the primary play shell should be sized so the full game area fits within a browser viewport of roughly 900 usable vertical pixels
- increasing the cell size for play readability is preferred over preserving non-essential chrome

User-facing HUD:

- displayed below the `NEXT` panel
- shows:
  - `SCORE`
  - score value
  - `PIECES`
  - piece count
- uses no debug-only state labels or internal counters
- in solid mode, the `NEXT` label may remain a simple helper label rather than a textured/noise-treated material role
- in noise mode, the `NEXT` label should not be shown

Debug HUD:

- exists only in debug boot modes
- may show any internal state useful for development
- is not part of the intended user presentation
- should include elapsed time since game start in milliseconds

Text rendering requirements:

- `SCORE` and `PIECES` are rendered as textured image assets rather than dynamic text
- `SCORE` and `PIECES` should be visually scaled to approximately the same height as the numeric digits
- numeric digits are rendered from textured digit assets rather than font rasterization
- user-HUD labels and numeric values should remain stable and readable rather than blinking
- the user HUD display caps at `99999` for score and `999` for pieces, but the underlying gameplay state is not capped by that presentation limit
- score uses a fixed five-digit right-aligned display region
- pieces uses a fixed three-digit right-aligned display region
- HUD textures and frame/panel materials should come from the active theme rather than being hard-coded in the renderer
- in noise mode, only the label and digit strokes should be drawn over the shared background; the renderer should not draw boxed HUD panels behind them
- in noise mode, HUD glyphs should use a higher white-sample probability than the gameplay surfaces so they remain readable at small size without turning into flat solid white text
- the current default noise-mode HUD glyph bias is `0.90` white probability versus `0.50` for the main gameplay-space noise surfaces

### 13. Settled Field Strategy

The settled field should conceptually be a composited surface.

This means:

- active pieces are drawn separately while falling
- when a piece locks, its visual material is merged into the settled field surface
- line clears operate on the settled field surface and/or its logical backing representation

This idea is useful in both modes.

In solid mode, it supports clean architectural symmetry with the final noise mode.

In noise mode, it is central to the concept: once the piece locks, it should become part of the same static visual world as everything else.

To preserve non-repeating tetromino materials after lock, settled cells should retain the source coordinates needed to continue sampling the same committed piece surface.

---

### 14. Noise Theme Principles

The noise theme must obey the following design rules:

- all major visible elements are built from fixed white-noise materials
- the noise is static unless the owning object moves or an explicit animation transforms it
- the noise should be composed of true black and true white samples rather than intermediate gray values
- the default grain should be specified in display-space units, currently `1 x 1` CSS-pixel blocks
- grain should be specified in display-space units rather than raw backing pixels so Retina-vs-non-Retina differences do not redefine the intended look
- the screen should not devolve into full-frame flicker chaos
- movement is the primary source of visibility
- stillness is concealment
- impact shake and line clear motion should briefly reveal structure
- if the player pauses on a frame, major scene elements should approach perceptual collapse rather than remaining separated by clear static chrome
- the effect should come from matched luminance and persistent material differences, not from full-frame regeneration

The effect depends on persistence.

If the white noise is regenerated every frame for static surfaces, the concept collapses into random flicker rather than perceptual concealment.

---

### 15. Solid Theme Principles

The solid theme is not a separate game design. It is a highly legible implementation of the same design.

Its purposes are:

- validating feel
- validating correctness
- debugging timing
- debugging movement and presentation behavior
- making theme-independent problems easy to see

The solid theme should use the same rendering pipeline, the same animation logic, and the same role structure as the noise theme.

It should differ only in assigned materials.

In the current implementation direction, the solid theme should also use generated persistent textures so that the renderer architecture stays aligned with the final noise mode.

---

### 16. Entry and Queue Behavior

The Next queue should be treated as part of the shared motion grammar of the game.

When a piece is consumed:

- the queue advances
- the visible previews slide upward as an animation
- a zero-duration option may be available for testing or alternate behavior

When a new piece becomes active:

- it should respect preloaded lateral input intent
- its entry should not feel detached from the player's buffered action

This matters especially in noise mode because the moment of entry is also a moment of visual discoverability.

---

### 17. Impact and Line-Clear Presentation

These are intentional customizations to the presentation model.

#### 17.1 Impact shake

When an active piece first makes grounded contact:

- the frame should shake
- the settled stack should shake
- the active piece should shake because it still exists at first-contact time
- the playfield surface and grid should remain fixed so the impact motion stays legible
- the effect should be vertical only; there should be no left-right shake component
- the motion should be a simple two-phase compression and return: first downward, then upward, then stop
- the initial downward compression should be stronger than the upward return so it reads as weight rather than generic vibration
- the effect should be short, strong enough to reveal structure, and tunable
- the current default impact profile is `8` frames with amplitude `8`

If an airborne hard drop locks immediately before a grounded frame can be displayed, it should still trigger the same impact shake.

The same active piece should not retrigger impact shake repeatedly just by briefly leaving and re-touching the surface.

In the current prototype, the player may also trigger the same shake manually with `S`. This is unrestricted and does not alter movement rules, but it does disable the `x2` no-shake score reward for the next line clear.

This is not merely decorative. It is part of how the noise concept becomes playable.

#### 17.2 Line-clear slide

When line(s) clear:

- the cleared row(s) should slide left beyond the frame before disappearing
- gameplay pauses in a controlled way during the animation
- the logical result after the animation remains a normal collapse of the field

Again, this is not theme-specific. The motion exists in both themes. The difference is that in noise mode it also serves a perceptual purpose.

---

### 18. Data Ownership

A clean implementation should separate ownership like this:

#### Gameplay core owns
- field occupancy grid
- active piece logical position and orientation
- queue contents
- timers and rule state
- grounded / lock state
- line-clear results

#### Presentation state owns
- visual transforms
- animation timers
- shake offsets
- queue slide offsets
- line clear visual offsets

#### Theme owns
- materials/textures for all visual roles
- any theme-specific palette or generation parameters

#### Renderer owns
- draw ordering
- compositing behavior
- presentation of the owned data above

This separation keeps the system understandable and keeps noise from infecting unrelated code.

---

### 19. Development Strategy

Development should proceed in this order:

1. Implement gameplay core with the solid theme
2. Tune feel until the game is satisfying as a falling-block game
3. Build the full shared presentation system
4. Refactor rendering around theme-owned persistent materials
5. Add the white-noise theme
6. Tune perceptual readability using motion, shake, and animation parameters

The white-noise concept should be present from the start as the target identity, but the engine should be proven in a readable theme before relying on perceptual difficulty.

---

### 20. Testing Philosophy

Testing should happen at two levels.

#### 20.1 Gameplay tests
- movement validity
- rotation validity
- grounded detection
- lock timing
- scoring correctness
- step-reset-only behavior
- state-machine transition order
- IRS and pre-entry lateral behavior
- hard drop immediate lock behavior
- queue consumption
- line clear correctness
- spawn blocking behavior
- piece-count-based gravity progression
- debug20g boot behavior
- `20G` input behavior with IRS, DAS preload, and active-phase manipulation

#### 20.2 Presentation tests
- queue slide behavior
- pre-entry lateral behavior
- impact shake timing
- line clear motion timing
- user HUD vs debug HUD separation
- textured HUD label / digit rendering behavior
- theme switching invariants
- visual compositing correctness

The solid theme should be used as the primary debugging view for presentation behavior before testing final readability in noise mode.

### 21. Definition of Success

The project succeeds when all of the following are true:

- the underlying game feels disciplined and high quality
- the engine behaves consistently across themes
- switching between solid and noise changes only materials, not gameplay
- the white-noise presentation makes motion informative and stillness concealing
- the white-noise theme feels like the identity of the project rather than a late gimmick

The desired end state is:

- technically, one coherent engine
- artistically, one unmistakably white-noise game

### 23. Short Summary

This project is a TAP/TGM2-ARS-inspired falling-block engine with a theme system whose artistic center is a white-noise presentation.

Its defining technical principle is that solid mode and noise mode must share the same gameplay, the same motion model, the same object model, and the same rendering structure.

Its defining artistic principle is that the white-noise theme transforms the same underlying game into a perception challenge where motion becomes information and stillness becomes concealment.
